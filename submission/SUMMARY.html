1 · The result

The baseline scores 2.3718 dev bpb. Three levers, each validated on the official scorer, take it to 1.7416 — a −26.6% reduction:

  1. Tokenizer (biggest lever). The corpus is ~33% Devanagari by bytes (every Hindi code point is 3 UTF-8 bytes), so the byte tokenizer spends 3 tokens per Hindi character. A byte-level BPE tokenizer trained only on the corpus compresses to 3.71 bytes/token, so the same 2,000 steps now cover ~a full epoch instead of <30% of the data. −0.46 bpb on its own.
  2. Trainer recipe. Higher LR (3e-4→1e-3) plus warmup + cosine decay: −0.16 bpb.
  3. Capacity allocation. Under the 2M cap, spending params on vocabulary/compression beat spending them on depth or width (vocab 3072/L3 > 2048/L3 > 1024/L4).

Key finding: the two “obvious” modern tweaks — AdamW with weight decay, and the textbook 0.02 init — both hurt. In a <1-epoch, 2,000-step regime the model is underfitting, so decay and small init slow the little learning the budget buys. Kept plain Adam + larger 0.05 init.

Phase 3 — ambitious swap: the Lion optimizer

Hypothesis: Lion (sign-momentum) is a fashionable Adam replacement — does it beat Adam on the best config?

optimizerlrdev bpbverdict
Adam (final)1e-31.7416best
Lion1e-3 (Adam's LR)2.0007lost badly
Lion (the fix)2e-41.9167recovered, still trails

Result & diagnosis: Lion lost. Its sign-based update applies a fixed step of magnitude ~lr regardless of the gradient, so a rate tuned for Adam (1e-3) is far too aggressive → 2.0007. The diagnosis-driven fix — drop the LR to 2e-4 — recovered most of the gap (→ 1.9167), confirming the mechanism, but Lion still trailed Adam: its edge shows up at large batch and long training, neither of which applies to a 2,000-step, batch-8 CPU run, where Adam's per-coordinate adaptivity simply fits faster. Kept Adam. (This is the "try something ambitious, watch it lose, explain why, fix the loss" the brief asks for — the full curve is in RUNLOG.md.)

4 · Why this is the best configuration

  • It attacks the scoring metric directly. bpb is per byte; the corpus’ Devanagari bytes are the expensive part, and BPE is the only lever that changes how many bytes each prediction covers. Compression is why the number moved most.
  • It respects the regime, not the folklore. The measured ladder — not intuition — showed that AdamW/decay/small-init hurt at <1 epoch; keeping Adam + 0.05 init is an evidence-based choice.
  • It spends the parameter cap where it pays. Tying + RoPE free up params (no separate output matrix, no learned position table) and every one goes into vocabulary, which the b1/b2/b3 sweep proved is the best use of the budget for this corpus.
  • It is honest and reproducible. Lossless tokenizer, official scorer for every number, and the unmodified grading command works as-is.

6 · Reproduce

# tokenizer is already trained (bpe.json). To retrain:
python tokenizer.py --corpus ../data/train_corpus.txt --vocab 3072 --out bpe.json

# final model (writes ckpt.pt with the recorded step count):
python train.py --data ../data/train_corpus.txt --dev ../data/dev_eval.txt \
  --steps 2000 --batch 8 --pos rope --block 256 --n_embd 192 --n_layer 3 \
  --n_head 4 --tie 1 --init_std 0.05 --opt adam --lr 1e-3 --schedule cosine \
  --warmup 100 --out ckpt.pt

# official score (unmodified interface):
python evaluate.py --checkpoint ckpt.pt --text_file ../data/dev_eval.txt
